# Project Brief

## Problema

Muchos materiales de ciencia de datos enseñan técnicas como si el negocio fuera un escenario limpio. En la práctica, las decisiones ocurren entre definiciones incompletas, plataformas limitadas, incentivos torcidos, ownership difuso y comités donde una gráfica puede ser técnicamente correcta pero organizacionalmente peligrosa.

Este proyecto enseña a detectar esas tensiones. Cada caso presenta una historia corporativa con datos sintéticos y una visualización mínima que revela una verdad incómoda. El estándar deseado no es "reporte con gráfica": es una investigación educativa breve donde el estudiante interpreta evidencia, sospecha de la historia oficial y aprende una regla transferible.

Este repositorio no crea dashboards. Crea narrativas con evidencia visual.

## Audiencia

- Alumnos de ciencia de datos aplicada.
- Analistas que necesitan explicar hallazgos a negocio.
- Equipos de Data que quieren practicar criterio, no solo técnica.
- Docentes que buscan casos con tensión realista y material listo para discusión.

## Casos De Uso

- Crear ejercicios de clase.
- Diseñar prácticas individuales.
- Revisar criterio analítico en entrevistas o bootcamps.
- Comparar soluciones débiles contra soluciones robustas.
- Enseñar cuándo no usar machine learning.
- Enseñar cuándo sí hace falta modelado, interpretabilidad, monitoreo o gobernanza.

## Principios

1. La narrativa manda; la gráfica prueba.
2. La técnica mínima suficiente gana.
3. La complejidad debe justificarse por el problema, no por el currículum del autor.
4. La mala decisión debe ser plausible, no caricaturesca.
5. El humor expone el absurdo organizacional, no reemplaza el análisis.
6. Los datos son sintéticos y mínimos.
7. El HTML es autocontenido, vertical y simple.
8. Ningún caso necesita filtros, pestañas ni paneles de KPIs.
9. La técnica mínima suficiente manda. No uses modelos si una tasa resuelve el caso. No uses una tasa si el comportamiento se adapta y requiere monitoreo.
10. La visualización es evidencia narrativa, no producto interactivo.
11. La intriga va antes que el contexto: el primer pantallazo debe provocar curiosidad, no explicar todo.
12. El alumno participa como investigador: mira pistas, formula hipótesis y descarta conclusiones cómodas.
13. Primero lenguaje humano, después término técnico.
14. El humor debe integrarse a la historia y cerrar con una consecuencia cómica o tragicómica.

## Qué Casos Queremos

- Casos donde una métrica se interpreta mal.
- Casos donde una definición induce conducta.
- Casos donde una ventana temporal oculta el fenómeno.
- Casos donde una solución cómoda empeora el problema.
- Casos donde una técnica simple revela más que un modelo innecesario.
- Casos donde un modelo ayuda, pero solo después de entender el negocio.
- Casos con conflictos entre Data, Negocio, Operativa, Tecnología, Riesgos, Legal o Auditoría.
- Casos que abren con una anomalía, contradicción o escena recordable.
- Casos donde la gráfica se lee como evidencia: pista, contraste, anotación, giro y consecuencia.
- Casos que terminan con una regla práctica que el alumno puede usar en otro problema.

## Qué Casos No Queremos

- Dashboards exploratorios.
- Herramientas self-service.
- Exploradores de datos.
- Aplicaciones de BI.
- Navegación compleja.
- Historias donde Data siempre tiene razón y el resto no entiende.
- Casos genéricos sin tensión organizacional.
- Modelos avanzados usados por decoración.
- Visualizaciones múltiples sin necesidad narrativa.
- Humor que ridiculiza personas o áreas.
- Datos reales, personales o sensibles.
- Reportes expositivos que empiezan con contexto y terminan con recomendaciones genéricas.
- KPI cards de monitoreo disfrazadas de tarjetas narrativas.
- Jerga técnica sin traducción humana previa.
